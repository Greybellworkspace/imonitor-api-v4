# Specification: Phase 3.9 — Background Jobs (Scheduler + Worker Scripts)

**Track ID:** phase3.9-background-jobs_20260313
**Type:** Feature
**Created:** 2026-03-13
**Status:** Draft

## Summary

Migrate the v3 cron scheduler (`startup.ts`) and its 6 associated worker scripts into the NestJS v4 architecture using `@nestjs/schedule` and `child_process`/`worker_threads`, matching v3 behavior exactly.

## Context

v3 runs all background jobs from `src/scripts/startup.ts`, which spawns `child_process.fork()` workers at configurable cron intervals. The workers connect to MySQL directly (outside InversifyJS DI). Three workers were already migrated in Phase 3.7 (`bulkProcess.worker.ts`, `cdrDecoder.worker.ts`, `billRun.worker.ts`). This phase migrates the remaining 6 workers and the scheduler module that orchestrates them.

## User Story

As the iMonitor system, I want background jobs to run on schedule so that automated reports are generated and delivered, old files/records are cleaned up, and observability alarms are evaluated every minute without requiring manual intervention.

## Acceptance Criteria

- [ ] `SchedulerModule` registers with `ScheduleModule.forRoot()` and is imported in `AppModule`
- [ ] 6 cron jobs implemented with correct schedules (matching v3 `startup.ts`), enabled only when `NODE_ENV !== 'test'`
- [ ] 6 worker scripts created in `src/scripts/worker/` (matching v3 logic): `automatedReport`, `automatedReportRetentionCleaning`, `scheduledBulkProcess`, `databaseRetentionCleanup`, `requestArchiveCleanup`, `observabilityAlarms`
- [ ] Cron schedule for `requestArchiveCleanup` is loaded dynamically from `core_sys_config` via `SystemConfigService` (key: `cleanUpCron`)
- [ ] All cron jobs guard against concurrent execution (v3 uses `processId` field on `core_automated_report` to mark in-progress rows)
- [ ] Unit tests for `SchedulerService` cover all 6 cron methods
- [ ] `npm run build`, `npm run lint`, `npm test` all pass

### Worker Scripts (new in Phase 3.9)

| Worker | Type | Purpose |
|--------|------|---------|

### File Locations

- Module: `src/modules/scheduler/`
- Workers: `src/scripts/worker/` (alongside existing 3.7 workers)

### Key v3 Patterns to Preserve

- Worker receives data via `process.on('message', ...)`, does work, calls `process.exit(0)` on success or `process.send({ error })` + `process.exit(1)` on failure
- Scheduler logs errors to `core_automated_report.errorStack` for AR workers
- `processId` checked before spawning to prevent double-execution
- Production-only guard: `if (process.env.NODE_ENV === 'production')` in v3 — preserve as `NODE_ENV !== 'test'` in v4 to allow staging runs
